01
Concurrency · Payment

Redis 멱등성 3중 방어선

"결제는 한 번만 성공해야 한다"
문제 원인

결제 요청은 사용자가 연타하거나, 네트워크 재시도가 걸리거나, 외부 PG 가 타임아웃 후 재시도하는 경우 같은 요청이 2번 들어온다. 이때 (1) 우리 시스템에서 결제가 2건 기록되고 (2) Toss 에서도 중복 승인될 경우 이중 과금이 발생한다. 클라이언트 debounce 만으로는 신뢰할 수 없고, 서버 단에서 요청 단위 멱등성을 강제해야 했다.

해결 과정

① AOP 기반 @Idempotent — 진입 차단

@Aspect
@Component
public class IdempotentAspect {

    private final RedisTemplate<String, String> redisTemplate;

    @Around("@annotation(idempotent)")
    public Object checkIdempotent(ProceedingJoinPoint jp, Idempotent idempotent) throws Throwable {
        String key = generateKey(jp, idempotent);
        Boolean success = redisTemplate.opsForValue()
            .setIfAbsent(key, "locked", idempotent.ttl(), TimeUnit.SECONDS);

        if (success == null || !success) {
            // 동일 키가 이미 있음 → 중복 요청
            throw new BusinessException(ErrorCode.DUPLICATE_REQUEST);
        }
        return jp.proceed();
    }
}
의사결정 근거
DB 유니크 제약으로도 중복은 막힌다. 하지만 그 시점이 트랜잭션 커밋 직전이라, 이미 Toss API 는 호출됐을 수 있다. AOP 로 컨트롤러 진입 순간에 Redis SETNX 로 락을 잡으면, 외부 호출 이전에 중복을 거를 수 있다. 즉 "어디서 막느냐" 에 따라 부작용 범위가 달라진다.

② PayRedisService.isIdempotent() — 서비스 계층 방어선

public boolean isIdempotent(String idempotencyKey) {
    String key = "idempotency-key:" + idempotencyKey;
    Boolean success = redisTemplate.opsForValue()
        .setIfAbsent(key, "processed", Duration.ofMinutes(5));
    return Boolean.TRUE.equals(success);
}

AOP 는 "같은 요청이 맞는가" 에 초점이 있다면, 여기는 "같은 결제 의도에 대한 고유키" (클라이언트가 생성) 를 5분간 잠근다. AOP 가 실패해도(Redis 장애, 키 생성 버그) 이 레이어에서 한 번 더 방어된다.

③ Toss Idempotency-Key + 자동 롤백

try {
    HttpHeaders headers = new HttpHeaders();
    headers.set("Idempotency-Key", idempotencyKey);  // Toss 자체 멱등성
    ResponseEntity<String> response = restTemplate.exchange(
        TOSS_CONFIRM_URL, HttpMethod.POST,
        new HttpEntity<>(body, headers), String.class);
    ...
} catch (Exception e) {
    // ✨ 실패 시 즉시 환불 — 돈이 떠있는 상태를 방치하지 않음
    payService.refundPayment(paymentKey, amount, "결제 검증 실패");
    throw e;
}
놓치기 쉬운 포인트
"우리 시스템에서 예외가 났다" = "Toss 에서도 실패했다" 가 아니다. Toss 는 승인이 성공했는데 우리 후처리에서 실패하는 경우가 있다. 이때 자동 환불 호출까지 설계해두지 않으면 "Toss 에는 돈이 있는데 우리 DB 엔 매칭권이 없음" 같은 사고가 난다.
결과
  • 결제 이중 과금 0건 (축제 운영 기간 전체)
  • 3중 방어로 하나의 레이어가 실패해도 다른 레이어가 보완 — depth-in-defense
  • 예외 시 자동 환불로 "돈이 떠있는 상태" 발생 차단

Business Logic · Matching

중요·일반 옵션 다단계 필터링

"매칭 성공률과 사용자 신뢰의 균형"
문제 원인

매칭에는 두 종류 조건이 있다. 중요 옵션(흡연/음주 등 절대 타협 불가) 과 일반 옵션(취미/학과 등 가능하면 맞으면 좋은 것). 단일 조건으로 필터링하면 매칭 실패율이 너무 높아지거나, 반대로 중요 조건을 어기는 매칭이 나오거나 둘 중 하나가 망한다. 실제 운영에서 이 균형을 어떻게 코드로 표현할지가 핵심 문제였다.

해결 과정

① 필터를 순서대로 적용하되, 통과 기준(minSize)을 옵션 타입별로 다르게

private List<Users> applyFiltersInOrder(
        List<Users> candidates,
        List<FilterCondition> filters) {

    List<Users> current = candidates;
    for (FilterCondition filter : filters) {
        List<Users> filtered = filter.apply(current);

        // 중요 옵션: 후보 1명만 남아도 진행 (절대 조건이므로)
        // 일반 옵션: 최소 3명 유지 (다양성 확보)
        if (filtered.size() < filter.minSize()) {
            break;  // 이 필터는 무시, 직전 상태 유지
        }
        current = filtered;
    }
    return current;
}

② 중요 옵션 실패 시 자동 환불

public void checkAndReturnImportantOptionPay(Users user, List<Users> pool) {
    if (user.getImportantOption() == null) return;

    boolean hasCandidate = pool.stream()
        .anyMatch(u -> matchesImportantOption(u, user.getImportantOption()));

    if (!hasCandidate) {
        // 중요 옵션에 맞는 후보가 0명 → 300원(중요 옵션 추가 비용) 자동 환불
        payService.refundPartial(user, 300L, "중요 옵션 후보 부재");
    }
}
의사결정 근거
중요 옵션은 돈을 더 받고 제공하는 기능이다. 후보 pool 이 작은 시간대(새벽) 에 "중요 옵션에 맞는 사람이 아예 없다"면, 차라리 무조건 매칭 실패 → 환불이 옳다. "비슷한 사람으로 매칭" 은 신뢰를 잃는 최악의 선택이다. 반면 일반 옵션은 취향 일치수준이므로 pool 을 너무 좁혀 매칭 실패를 만드는 것보다 완화가 낫다. minSize 1 vs 3 은 이 철학의 코드 표현이다.
결과
  • 중요 옵션 위반 매칭 0건 — 사용자 신뢰 유지
  • 일반 옵션은 완화 적용으로 매칭 실패율 감소
  • 중요 옵션 실패 시 자동 환불로 고객 문의/분쟁 제로
03
Concurrency · Locking

낙관적 락 재시도 + 비관적 락 병행

"동시성 전략은 데이터 특성에 따라 다르다"
문제 원인

한 사용자가 여러 매칭 요청을 동시에 보내거나, 인기 많은 사용자(enemy) 의 pickedCount 가 동시에 증가할 때 경합이 생긴다. 모든 경로에 비관적 락만 쓰면 처리량이 떨어지고, 낙관적 락만 쓰면 실패율이 올라간다. 데이터 특성별로 전략을 분리해야 했다.

해결 과정

① 핫한 카운터는 낙관적 락 + retry (충돌 희귀한 영역)

@Transactional
public void updateEnemyPickedCount(Users enemy) {
    int retryCount = 3;
    while (retryCount-- > 0) {
        try {
            Users latest = usersRepository.findById(enemy.getId()).orElseThrow();
            latest.increasePickedCount();
            usersRepository.save(latest);
            return;
        } catch (OptimisticLockingFailureException e) {
            if (retryCount == 0) throw e;
            // 다음 루프에서 최신 버전으로 재시도
        }
    }
}

② 결정적 갱신은 비관적 락 (절대 놓칠 수 없는 영역)

public interface UsersRepository extends JpaRepository<Users, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT u FROM Users u WHERE u.uuid = :uuid")
    Optional<Users> findUsersByUuidForUpdate(@Param("uuid") String uuid);
}
의사결정 근거
pickedCount 는 "+1 한 번 더 해도 되지만, 놓치면 아쉬운 정도"다. 충돌이 적고, 재시도 3회로 대부분 복구된다 → 낙관적 락이 처리량에 유리. UUID 로 찾는 사용자 본체는 결제/매칭 직전 상태 변경이 걸린 데이터다. 재시도가 실패하면 사용자 경험이 깨진다 → 비관적 락으로 직렬화. "두 전략 중 하나를 고른다" 가 아니라, "영역마다 맞는 걸 고른다".
결과
  • 핫 카운터 업데이트는 낙관적 락 + 재시도 3회로 대부분 성공
  • 결제/매칭 경로의 본체 User 는 비관적 락으로 경합 직렬화
  • 축제 운영 중 동시성 관련 장애 0건
Key Takeaways
01
다중 방어선이 안전을 만든다 — AOP / 서비스 계층 / 외부 API 자체 멱등성까지 3단으로 겹쳐두면 하나가 실패해도 다른 레이어가 잡는다.
02
환불 조건도 설계의 일부다 — "실패 시 어떻게 상태를 되돌릴 것인가" 를 정상 로직과 함께 설계해야 사용자 신뢰가 생긴다.
03
동시성 전략은 영역별로 다르다 — 모든 곳에 비관적 락이면 처리량이 죽고, 모든 곳에 낙관적 락이면 결정적 경로가 깨진다. 데이터 특성을 읽고 나눠 쓰는 게 맞다.
