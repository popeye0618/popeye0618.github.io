01
MSA Communication · Hybrid

Kafka + OpenFeign 하이브리드 통신

"성격에 따라 통신 방식을 나눈다"
문제 원인

v3(모놀리식) → v5(MSA) 로 분리하자마자 상반된 문제가 동시에 터졌다. 정합성 깨짐 — user-service 에서 프로필을 수정해도 matching-service 의 후보 테이블은 예전 값이라 잘못된 매칭이 나왔다. 그렇다고 매번 동기 조회하면 매칭 한 번에 수백 번의 내부 호출이 발생. 반대로 매칭 상대의 최신 프로필을 화면에 띄울 땐 로컬 복제본으로는 안 되고 즉시 일관성이 필요했다. 단일 통신 방식으로는 둘 다 풀 수 없었다.

해결 과정

① Kafka — 최종 일관성 경로 (로컬 복제본)

// matching-service/infra/kafka/ProfileEventConsumer.java
@KafkaListener(topics = "profile-updates", groupId = "matching-service-group")
public void consumeProfileUpdate(ProfileUpdatedMatchingEvent event) {
    candidateService.upsertCandidate(event);
}

matching-service 가 자기만의 MatchingCandidate 테이블을 갖고, 검색/필터링/스코어링은 자기 DB 안에서만 한다 → 매칭 한 번에 수백 번의 내부 호출 제거. 토픽을 member-signup / member-update / profile-updates / matching-success 등 성격별로 분리하고, groupId 를 구독자별로 분리해 오프셋 추적을 격리했다.

② OpenFeign — 실시간 조회 경로

// matching-service/infra/client/MemberClient.java
@FeignClient(name = "user-service",
             path = "/api/internal/users",
             url  = "${user-service.url}")
public interface MemberClient {
    @GetMapping("/{memberId}/profile")
    ProfileResponse getProfile(@PathVariable Long memberId);
}
의사결정 근거 — "1~2초 늦어도 사용자가 알아차릴까?"
판단 기준은 이거 하나였다. 최종 일관성으로 충분한 경로는 Kafka (프로필 변경 → 매칭 후보 테이블), 즉시 일관성이 필요한 경로는 OpenFeign (매칭 결과 화면의 상대방 프로필). 내부 전용 엔드포인트 /api/internal/users/ 는 Gateway 를 거치지 않아 JWT 재검증 오버헤드도 제거했다.

③ 통신 방식 선택 기준 (실제로 내린 판단들)

Kafka — 프로필 업데이트 → 매칭 후보 (지연 허용), 회원 탈퇴 → 매칭 후보 제거 (fan-out), 매칭 성공 알림 (수신자 다수).
Feign — 매칭 결과 화면의 상대방 프로필 (0초 지연), 포인트 차감 직전 잔액 검증 (원자성 중요).
결과
  • 정합성 유지 + 성능 확보: 매칭 1회당 내부 호출 수백 번 → 0~1회
  • Kafka 경로는 user-service 장애 시에도 matching-service 가 로컬 DB 로 계속 동작
  • 새 서비스가 붙어도 profile-updates 토픽 구독 한 줄로 연결
02
MSA Boundary · Reflective Refactoring

MSA 경계 재설정 — 쪼갰다가 다시 합쳤다

"MSA 는 정답이 아니라 선택이다. 틀렸으면 되돌려야 한다."
문제 원인

초기 설계는 auth-service / member-service / profile 을 각자 독립 서비스로 분리했다. 처음엔 깔끔해 보였다. 하지만 1년간 운영하며 드러난 비용은 달랐다. 회원가입 플로우에서 분산 트랜잭션 지옥이 발생했고 (auth 커밋 → member 실패 시 Saga 복잡도가 본 로직을 역전), Feign 호출이 서로를 7번 호출하는 병목으로 쌓였으며, DDD 관점에서 두 서비스의 도메인 경계는 사실상 같았다. MSA 는 서비스 간 호출 비용이 도메인 분리 이점보다 크면 실패다.

해결 과정

① 통합 범위 확정

auth-service + member-service → user-service 하나로 통합. matching-service, chat-service, item-service 는 도메인이 명확히 달라 유지. "서비스 수가 적으면 좋다" 가 아니라 "경계가 도메인과 맞는 만큼" 이 기준.

② Feign 호출 7개 → 0개

// Before — member-service → auth-service
@FeignClient(name = "auth-service", ...)
public interface AuthServiceClient {
    Boolean isVerified(Long memberId);   // 네트워크 왕복
}

// After — user-service 내부
@Service
public class AuthService {
    public boolean isVerified(Long memberId) { ... }  // 같은 JVM, 로컬 호출
}

③ 트랜잭션 경계 단순화 — 분산 → 단일

// Before: 분산 트랜잭션
// [auth-svc] 인증정보 저장 → commit
//   ↓ Feign 호출 (네트워크 경계)
// [member-svc] 회원/프로필 저장 → commit
//   ↓ 실패 시? auth-svc 보상 트랜잭션 필요 (Saga)

// After: 단일 트랜잭션
@Transactional
public void signup(SignupRequest req) {
    AuthCredential auth    = authService.create(req);
    Member         member  = memberService.create(auth, req);
    Profile        profile = profileService.create(member, req);
    // 어느 하나라도 실패하면 전체 롤백 — 표준 JPA 트랜잭션
}
의사결정 근거
같은 도메인끼리 Feign 은 결합도는 그대로인데 성능만 희생하는 구조였다. Saga 복잡도 > MSA 이점 이라면 되돌리는 게 맞다. 외부 서비스(matching / notification)는 여전히 이벤트가 필요하므로 Kafka 토픽 4개는 유지 — 도메인이 다른 서비스와의 비동기 연결은 계속 필요하기 때문.
결과
  • Feign 호출 7개 → 0개 (네트워크 왕복 완전 제거)
  • 회원가입이 표준 @Transactional 하나로 정리 · Saga 보상 로직 삭제
  • 응답 시간 개선 예상치 ~60% (네트워크 오버헤드 제거 분)
  • 러닝 포인트 — "MSA 를 언제 도입하고 언제 거둘지" 를 실제로 경험
03
Caching · Performance

채팅 차단 Redis 캐싱 · 9배 성능

"쓰기 드물고 읽기 많은 데이터 — 전형적 캐시 지점"
문제 원인

채팅 메시지 하나가 전송될 때마다 isBlocked(sender, receiver) 호출이 반드시 필요하다. 활성 채팅방 1개 = 초당 10개 메시지 기준 1분 = 600번의 DB 조회. 채팅방이 N 개면 × N 배. 서비스가 커질수록 병목이 선형 증가한다. 차단 목록은 쓰기는 드물고 읽기는 매우 빈번 — 전형적인 캐시 적용 지점이지만, 보안 관련 데이터라 정합성이 깨지면 차단된 유저의 메시지가 새어 들어간다.

해결 과정

① 캐시 키/구조 · Look-Aside 패턴

private static final String BLOCK_CACHE_KEY_PREFIX = "user:block:";
private static final Duration CACHE_TTL = Duration.ofMinutes(30);

@Override
@Transactional(readOnly = true)
public Set<Long> getBlockedUserIds(Long blockerUserId) {
    String cacheKey = BLOCK_CACHE_KEY_PREFIX + blockerUserId;
    Object cachedValue = redisTemplate.opsForValue().get(cacheKey);

    if (cachedValue != null) {
        // Redis JSON 직렬화 호환: Set 이 List 로 돌아올 수 있음
        if (cachedValue instanceof List) return new HashSet<>((List<Long>) cachedValue);
        if (cachedValue instanceof Set)  return (Set<Long>) cachedValue;
    }

    // 캐시 미스 → MongoDB 조회 후 적재
    Set<Long> blockedUserIds = userBlockRepository.findByBlockerUserId(blockerUserId).stream()
        .map(UserBlock::getBlockedUserId)
        .collect(Collectors.toSet());
    redisTemplate.opsForValue().set(cacheKey, blockedUserIds, CACHE_TTL);
    return blockedUserIds;
}

키 구조는 차단 주체 기준 (user:block:{blockerUserId}) 으로 뒀다. 값은 Set<Long> 이라 contains() 가 O(1). TTL 30분은 "차단 해제 후 최악의 경우라도 30분 안에 반영" 의 보장선 — 너무 짧으면 캐시 효과가 사라지고, 너무 길면 invalidate 실패 시 위험이 커진다.

② 캐시 무효화 — 차단/해제 시 즉시 삭제

public void blockUser(Long blockerUserId, Long blockedUserId) {
    ...
    userBlockRepository.save(userBlock);
    evictBlockCache(blockerUserId);  // ✅ 즉시 삭제
}

public void unblockUser(Long blockerUserId, Long blockedUserId) {
    ...
    evictBlockCache(blockerUserId);  // ✅ 즉시 삭제
}
의사결정 근거 — TTL + 즉시 무효화 이중 안전장치
TTL 만으로 끝내면 차단 해제 후에도 30분간 차단 상태 유지 → UX 문제. 즉시 무효화 만으로 끝내면 Redis 장애 시 영원히 stale → 안전 문제. 둘 다 적용: 정상 상태에서는 즉시 반영, Redis 장애 시에도 최대 30분 뒤 자연 복구. 하나만 쓰는 건 둘 중 하나의 실패 시나리오를 방치하는 것과 같다.
결과 — JUnit 성능 테스트로 정량 검증

│ 호출 횟수   │  미적용    │  적용    │  향상     │
├─────────────┼───────────┼──────────┼───────────┤
│   100 회    │  1,242ms  │   144ms  │   8.6 ×   │

// 실서비스 시나리오 (1분 · 초당 10개 메시지 = 600건)
//   누적 지연    : 7.3초 → 0.8초   (9.3배 향상)
//   DB 쿼리 수   :  600  →   1     (99.8% 감소)
  • DB 쿼리 99.8% 감소 — MongoDB 부하 거의 제거, 채팅방이 100배 늘어나도 여유
  • 호출 수가 늘수록 배율이 올라감 (8.6 → 9.3 → 9.5) — 스케일링에 유리한 특성
  • 실시간 채팅 UX 핵심인 "즉시성" 을 지킬 수 있는 구조
Key Takeaways
01
통신 방식은 "성격에 따라" 나눈다 — 최종 일관성 OK 면 Kafka, 즉시성 필수면 OpenFeign. 모든 경로를 한 방식으로 통일하는 건 설계 게으름이다.
02
MSA 는 언제든 되돌릴 수 있어야 한다 — 7개의 Feign 호출과 분산 트랜잭션이 보이면 도메인 경계를 잘못 그었다는 신호. "쪼갰다가 합쳐본" 경험이 진짜 MSA 역량이다.
03
캐시는 "즉시 무효화 + TTL" 이중 안전장치 — 하나만으로는 UX 실패 또는 장애 전파. 두 축을 함께 둘 때 비로소 안정적 캐시다.
