01
Async Pipeline · AWS MediaConvert

MediaConvert 비동기 변환 파이프라인

"변환은 AWS 가, 상태 추적은 우리가"
문제 원인

브라우저 MediaRecorder 는 .webm(VP8/VP9) 을 출력하지만 iOS Safari 등에서 재생이 제한돼 .mp4(H.264) 변환이 필수였다. 처음엔 서버에서 FFmpeg 로 직접 변환했는데 두 가지가 치명적이었다. 동기 blocking — 네컷 기준 수 초~수십 초 걸리는 변환이 Tomcat 스레드를 묶어 다른 요청이 밀렸고, 서버 자원 고갈 — FFmpeg 의 CPU 부하가 전체 API 응답 시간을 끌어내렸다. "인코딩을 내가 할 이유가 없다" 는 결론에 도달했다.

해결 과정

① 책임 경계 재설계 — taskId 와 jobId 분리

// TranscodingService.java — 요청 / 완료를 분리
public TranscodeTaskSubmitResponse submitTranscodeTask(String userPublicId, String fileName) {
    String taskId = UUID.randomUUID().toString();
    LocalDateTime now = LocalDateTime.now();

    // ① 우리 시스템의 taskId 로 상태 먼저 기록 (QUEUED)
    TranscodeTaskState queuedState = TranscodeTaskState.queued(taskId, userPublicId, fileName, now);
    transcodeTaskRepository.save(queuedState);

    // ② AWS 에 Job 생성 요청 → MediaConvert jobId 획득
    String jobId = createConversionJob(userPublicId, fileName);

    // ③ 두 ID 를 bidirectional 매핑 (콜백에서 taskId 역추적용)
    TranscodeTaskState submittedState = queuedState.withSubmitted(jobId, LocalDateTime.now());
    transcodeTaskRepository.save(submittedState);
    transcodeTaskRepository.linkJobToTask(jobId, taskId);

    return new TranscodeTaskSubmitResponse(taskId, jobId, submittedState.status(), ...);
}
의사결정 근거 — 왜 taskId 와 jobId 를 분리했는가?
jobId 는 AWS 가 발급하는 외부 식별자다. 이걸 그대로 사용자에게 노출하면 (1) AWS 인프라가 교체되는 순간 API 계약이 깨지고 (2) AWS 요청이 실패한 시점부터 완료까지 사이에는 jobId 가 존재하지 않아 클라이언트에게 돌려줄 핸들이 없다. 우리 시스템의 taskId 를 먼저 발급하고 AWS jobId 와 매핑하면, AWS 요청 실패도 "QUEUED 로 기록된 task 가 제출 실패했다" 는 식으로 일관되게 표현된다.

② 상태 모델 — immutable record + 전이 메서드

public record TranscodeTaskState(
    String taskId, String userPublicId, String originalFileName,
    String jobId, TranscodeTaskStatus status, String errorMessage,
    UserMediaResponse media, LocalDateTime createdAt, LocalDateTime updatedAt
) {
    public static TranscodeTaskState queued(...)                      { ... }
    public TranscodeTaskState withSubmitted(String jobId, LocalDateTime now)   { ... }
    public TranscodeTaskState withProgressing(LocalDateTime now)               { ... }
    public TranscodeTaskState withComplete(UserMediaResponse m, LocalDateTime n) { ... }
    public TranscodeTaskState withError(String message, LocalDateTime now)     { ... }
}

상태 전이(QUEUED → SUBMITTED → PROGRESSING → COMPLETE | ERROR)를 각 with* 메서드로 강제. 부분 업데이트를 허용하지 않고, 전이별로 어떤 필드가 바뀌는지 타입이 강제한다 — withError 는 errorMessage 필수, withComplete 는 media 필수. 잘못된 전이는 컴파일 타임에 차단된다.

③ 저장소는 Redis — TTL 7일 자연 소멸

private static final String TASK_KEY_PREFIX = "media:transcode:task:";
private static final String JOB_KEY_PREFIX  = "media:transcode:job:";
private static final Duration TASK_TTL = Duration.ofDays(7);

@Override
public void linkJobToTask(String jobId, String taskId) {
    // jobId → taskId 역인덱스 (SNS 콜백은 jobId 만 전달)
    redisTemplate.opsForValue().set(JOB_KEY_PREFIX + jobId, taskId, TASK_TTL);
}

변환 작업 상태는 수명이 짧고 (완료되면 UserMedia 로 영속화), 조회는 빈번 (폴링). 이 특성에 RDB 는 과하다. Redis 는 TTL 7일 후 자동 정리 → 좀비 레코드 관리 비용 제로, 단일 GET 으로 역인덱스 조회, 쓰기 지연이 거의 없어 상태 전이에 적합.

④ SNS Webhook — 변환 성공했는데 DB 저장 실패 시나리오

private void processConversionResult(String messageJson) {
    String state = detail.path("status").asText("UNKNOWN");
    if ("COMPLETE".equals(state)) {
        try {
            transcodingService.handleCompletedJob(jobId, userPublicId, originalFileName, outputS3Path);
        } catch (Exception e) {
            // ✨ DB 저장 실패 시 ERROR 로 전이 — 완료로 잘못 커밋하지 않음
            transcodingService.handleFailedJob(jobId, "Failed to persist transcoded video");
        }
    } else if ("ERROR".equals(state)) {
        transcodingService.handleFailedJob(jobId, errorMessage);
    }
}
놓치기 쉬운 포인트 — "소비자 관점의 진실"
AWS 는 COMPLETE 를 쐈지만 우리 쪽 DB 저장이 실패할 수 있다 (커넥션 오류 등). 이때 클라이언트 관점에선 "변환은 끝났는데 내 영상이 목록에 안 뜬다" 는 모순이 생긴다. 그래서 catch 에서 handleFailedJob 으로 전이시켜 일관성을 유지했다 — "소비자 관점의 진실(내 영상이 보이느냐)" 을 기준으로 상태를 정렬한 것.
결과
  • Tomcat 스레드가 인코딩 작업에 묶이지 않음 — API 응답은 수백ms, 인코딩(수십 초)은 AWS 전담
  • 서버 CPU 급등 해소, 동시 변환 처리 한도를 AWS 스케일링에 위임
  • taskId ↔ jobId 매핑으로 AWS 인프라 변경이 API 계약에 영향 없음
02
Strategy Pattern · Presigned URL

S3 업로드 Strategy + Presigned URL

"업로드 경로가 6종이고, 앞으로도 늘어난다"
문제 원인

업로드 타입이 6종으로 불어났다 — PROFILE / FRAME / FRAME_COMPONENT / FOURCUT_PHOTO / FOURCUT_VIDEO. 각 타입마다 S3 경로, 파일명, 임시/영구 구분이 조금씩 달랐다. 초기엔 FileService 한 곳에서 switch-case 로 분기했는데 타입이 추가될 때마다 메서드가 계속 길어지고 특정 타입만 격리 테스트가 어려웠다. 더 큰 문제는 업로드 방식 — "파일 → 서버 → S3" 라우팅으로 받다보니 네컷 영상(수십 MB) 이 서버 대역폭 · 메모리를 크게 먹었다.

해결 과정

① 업로드 방식을 Presigned URL 로 전환

@Override
public PresignedUploadResponse generatePresignedUploadUrl(
    UploadType uploadType, String originalFilename,
    ContentType contentType, String publicId, boolean isTemp
) {
    // MIME-확장자 일관성 검증 (image/png 인데 .mp4 를 보내는 위조 차단)
    ContentType validated = ContentType.validate(contentType.getMimeType(),
                                                 extractExtension(originalFilename));

    // Strategy 선택
    UploadPathStrategy strategy = strategyMap.get(uploadType);
    if (strategy == null) throw new BusinessException(StorageErrorCode.UNSUPPORTED_UPLOAD_TYPE);
    String key = strategy.generateKey(publicId, originalFilename, isTemp);

    PutObjectRequest putObjectRequest = PutObjectRequest.builder()
        .bucket(bucketName).key(key)
        .contentType(validated.getMimeType())   // 서명에 MIME 타입까지 묶음
        .build();

    PresignedPutObjectRequest presigned = s3Presigner.presignPutObject(
        PutObjectPresignRequest.builder()
            .signatureDuration(EXPIRY)          // 1일
            .putObjectRequest(putObjectRequest)
            .build()
    );
    return new PresignedUploadResponse(key, presigned.url().toString(), ...);
}
Presigned URL 의 보안 이중 방어선
PUT URL 만 내주는 게 아니라 PutObjectRequest 자체에 contentType 까지 묶어서 서명했다. 이러면 클라이언트가 다른 MIME 으로 올리면 S3 가 거부한다. 여기에 ContentType.validate(mime, ext) 로 "image/png 인데 확장자 .exe" 같은 헤더-확장자 불일치를 사전 차단 — 이중 방어선. URL 유효기간도 1일로 짧게 제한해 탈취 시 영향 범위를 제한했다.

② 경로 규칙을 Strategy 패턴으로 분리

public interface UploadPathStrategy {
    UploadType getUploadType();
    String generateKey(String publicId, String originalFilename, boolean isTemp);
}

@Component
public class ProfileUploadPathStrategy implements UploadPathStrategy {
    public UploadType getUploadType() { return UploadType.PROFILE; }
    public String generateKey(String publicId, String originalFilename, boolean isTemp) {
        String uniqueName = UUID.randomUUID() + extractExtension(originalFilename);
        return String.format("uploads/users/%s/profile/%s", publicId, uniqueName);
    }
}

@Component
public class FourcutVideoUploadPathStrategy implements UploadPathStrategy {
    public UploadType getUploadType() { return UploadType.FOURCUT_VIDEO; }
    public String generateKey(String publicId, String originalFilename, boolean isTemp) {
        return String.format("uploads/users/%s/webm/%s",
                             publicId, UUID.randomUUID() + extractExtension(originalFilename));
    }
}

③ Spring DI 로 Map 자동 조립

public S3FileStorageService(
    S3Client s3Client, S3Presigner s3Presigner,
    Set<UploadPathStrategy> strategies,   // Spring 이 모든 구현체를 자동 수집
    String bucketName
) {
    this.strategyMap = strategies.stream()
        .collect(Collectors.toMap(UploadPathStrategy::getUploadType, Function.identity()));
}
의사결정 근거 — 왜 Strategy 패턴인가 (switch 로도 되는데)
Strategy 가 @Component 로 빈이 되면 (1) 다른 서비스(MediaConvert 출력 경로 등)에서도 주입받아 쓸 수 있고 (2) 새 업로드 타입 추가 시 기존 코드를 열지 않아도 되어 OCP 만족. (3) 각 Strategy 를 독립적으로 유닛 테스트할 수 있어 회귀 버그 방지. 경로 규칙이 상품팀 요청으로 자주 바뀌는 영역이라 효과가 컸다.
결과
  • 서버가 업로드 트래픽을 경유하지 않음 — 수십 MB 영상도 서버 대역폭/메모리 부담 없음
  • 새 업로드 타입 추가 = Strategy 클래스 1개 + UploadType enum 값 1개 (기존 코드 무수정)
  • MIME-확장자 일관성 검증 + 서명에 ContentType 포함으로 위조 업로드 이중 방어
03
AOP · Availability vs Consistency

중복 요청 차단 AOP · FAIL_OPEN / FAIL_CLOSE

"Redis 가 죽었을 때 서비스도 같이 죽일 것인가?"
문제 원인

이메일 인증, 회원가입, 프레임 업로드 등 일부 엔드포인트는 연타되면 비용이 발생하거나 일관성이 깨진다. 클라이언트 debounce 만으로는 신뢰할 수 없어 서버 단에서 멱등성을 강제해야 했다. AOP + Redis SETNX 는 전형적 해결책이지만 한 가지 질문이 따라붙었다 — "Redis 가 죽었을 때 정책이 다르다". 이메일 인증 코드는 차라리 발송을 허용해야 (가입 봉쇄 방지), 결제/회원가입은 차라리 차단해야 (데이터 오염 방지). 엔드포인트마다 다른 정책을 어노테이션 하나로 표현할 수 있어야 했다.

해결 과정

① 어노테이션에 전략을 선언적으로 명시

public enum LockStrategy {
    FAIL_CLOSE,   // Redis 장애 시 요청 차단 (가용성 < 일관성)
    FAIL_OPEN     // Redis 장애 시 요청 허용 (가용성 > 일관성)
}

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface PreventDuplicateRequest {
    String key()            default "";                                   // SpEL 식
    long   time()           default 3000;                                 // 락 유지 시간
    TimeUnit timeUnit()     default TimeUnit.MILLISECONDS;
    LockStrategy strategy() default LockStrategy.FAIL_CLOSE;             // 기본은 안전한 쪽
}
의사결정 근거 — 왜 default 가 FAIL_CLOSE 인가?
"무조건 기본값이 안전한 쪽" 이라는 원칙을 따랐다. 개발자가 전략을 잊어버리거나 신입이 어노테이션을 새로 붙여도 일관성이 깨지지 않는 방향이 기본값이어야 한다. FAIL_OPEN 은 명시적으로 의도를 선언해야만 활성화된다.

② AOP 에서 전략 분기

@Around("@annotation(preventDuplicateRequest)")
public Object checkDuplicate(ProceedingJoinPoint joinPoint,
                             PreventDuplicateRequest preventDuplicateRequest) throws Throwable {
    String uniqueKey = generateUniqueKey(joinPoint, preventDuplicateRequest.key());
    String redisKey  = "duplicate:request:" + uniqueKey;

    try {
        // ① SETNX 로 선착순 락 획득
        Boolean isSuccess = redisTemplate.opsForValue()
            .setIfAbsent(redisKey, "locked", Duration.ofMillis(unit.toMillis(time)));

        if (isSuccess == null || !isSuccess) {
            // ② 이미 누가 잡고 있음 → 중복 차단 (전략과 무관, 항상 차단)
            throw new BusinessException(GlobalErrorCode.DUPLICATE_REQUEST);
        }
    } catch (Exception e) {
        if (e instanceof BusinessException) throw e;

        // ③ 여기면 Redis 자체가 맛이 간 상황
        if (strategy == LockStrategy.FAIL_OPEN) {
            log.error("Redis 연결 실패 - Fail Open 정책에 의해 로직 실행 허용");
            // 로직 계속 진행
        } else {
            throw new BusinessException(GlobalErrorCode.REDIS_CONNECTION_ERROR);
        }
    }
    return joinPoint.proceed();
}

첫 번째 throw(DUPLICATE_REQUEST) 는 catch 에서 재던짐 (instanceof 체크). 두 번째 throw(REDIS_CONNECTION_ERROR) 는 완전히 다른 원인 — "너무 빨리 다시 눌렀어요" 와 "서버 인프라 문제" 는 메시지 · 재시도 전략이 달라야 한다. 에러 코드를 분리해 프론트에서 사용자 안내 문구와 재시도 정책을 구분할 수 있게 됐다.

③ SpEL 기반 동적 키 + 엔드포인트별 차등 적용

// MailAuthController — 이메일 인증 발송: FAIL_OPEN
@PreventDuplicateRequest(key = "#request.email", strategy = LockStrategy.FAIL_OPEN)
@PostMapping("/code")
public ResponseEntity<Response<Void>> sendAuthCode(@RequestBody @Valid EmailAuthRequest request) {
    mailAuthService.sendAuthCode(request.email());
    return ResponseEntity.ok(Response.success());
}
의사결정 근거 — 이메일 인증에 FAIL_OPEN 을 선택한 이유
이메일 인증은 "가입 전 단계" 다. 여기서 Redis 장애로 요청이 차단되면 신규 가입 자체가 막힌다 — 서비스 출발점이 봉쇄되는 것. 반면 장애 중 연타로 이메일이 2~3번 가는 것은 (a) 사용자 불편은 "한 번 더 받아서 무시" 수준이고 (b) SMTP 서버 Rate Limit 을 따로 걸면 과금 위험도 제한된다. "기반 시설 장애가 핵심 사용자 여정을 봉쇄하는 것보다 낫다" 는 trade-off.
결과
  • 어노테이션 한 줄로 중복 방지 + SpEL 키 + 장애 시 정책 을 선언적으로 표현
  • FAIL_OPEN / FAIL_CLOSE 를 엔드포인트별로 차등 적용 — 가용성/일관성 trade-off 가능
  • 핵심 사용자 여정이 Redis 장애로 전체 봉쇄되는 최악 시나리오 방지
  • DuplicateRequestAspectTest 로 SETNX 실패 · Redis 예외 · SpEL 추출 케이스 분리 검증
Key Takeaways
01
비동기 외주화 설계 — 서버가 직접 하지 않아도 되는 작업(인코딩)을 AWS 에 위임하고, taskId ↔ jobId 매핑으로 우리 도메인 모델의 정합성을 지켰다.
02
Strategy 패턴 + DI — 업로드 경로 규칙을 확장에 열려있고 변경에 닫힌(OCP) 구조로 분리해, 새 업로드 타입 추가가 기존 코드에 영향을 주지 않게 했다.
03
