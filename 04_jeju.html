01
Spec Layer · CLAUDE.md

CLAUDE.md 2단 계층화

"AI 는 당신이 알려준 만큼만 안다"
문제 원인

Claude 같은 LLM 에이전트를 Spring 프로젝트에 투입하면 초반 생산성이 폭발하지만 곧 특정 패턴의 반복적 실수가 나타난다. Controller 에서 Response wrapper 없이 객체를 그대로 반환하고, 에러 상황에 맞는 에러코드를 매번 새로 창작하고 (USER_NOT_FOUND → USER_001 → NOT_FOUND_USER), HTTP 상태 코드를 감에 의존한다. 이건 AI 의 한계가 아니라 내가 AI 에게 규칙을 알려주지 않은 것이 원인이다. 프로젝트 관례를 "파일 곳곳에서 유추하라" 고 방치하면, AI 는 대안 중 아무거나 선택한다.

해결 과정

① 루트 CLAUDE.md — "이 프로젝트는 무엇인가"

## Build Commands

## API Specification
> 백엔드 작업 시 `backend/CLAUDE.md` 참조
의사결정 근거 — 왜 2단으로 분리했는가?
단일 파일로 두면 (1) 프론트 작업 중일 때도 백엔드 규칙까지 전부 컨텍스트에 올라가 노이즈가 커지고 (2) 규칙이 늘수록 파일이 비대해져 실제로 AI 의 준수율이 떨어지는 현상이 보였다. "얕은 전체 맥락(루트) + 깊은 도메인 규칙(하위 디렉토리)" 구조 로 두면 AI 는 작업 맥락에 맞는 문서만 집중해서 읽는다. 프로젝트가 커져도 확장 가능한 구조.

② backend/CLAUDE.md — "백엔드 작업 시 반드시 지켜야 할 것"

# Backend CLAUDE.md

## API Response Structure
모든 응답은 아래 wrapper 를 **반드시** 사용. Controller 에서 직접 객체 반환 **금지**.
{ "success": true, "code": "SUCCESS", "message": "...", "data": {} }

## HTTP Status Rules
| 200 | 조회·수정·삭제 성공                              |
| 201 | 리소스 생성                                     |
| 400 | Bean Validation 실패 (COMMON_001)               |
| 401 | 인증 오류 (AUTH_001~004)                        |
| 403 | 권한 없음 (ITIN_002)                             |
| 404 | 리소스 없음 (USER_002, ITIN_001, PLACE_001)     |
| 409 | 중복 (USER_001, REVIEW_001, REPORT_001)         |
| 500 | 서버 오류 (COMMON_002)                          |
| 502 | 외부 API 오류 (PLACE_002)                        |

## Error Codes (총 15개)
| COMMON_001 | 유효하지 않은 입력값            |
| AUTH_001   | 이메일/비밀번호 불일치          |
| AUTH_002   | 이메일 미인증                   |
| ITIN_001   | 존재하지 않는 일정              |
| PLACE_002  | 한국관광공사 API 조회 실패      |
| ...
놓치기 쉬운 포인트 — "강한 명령형" 으로 쓰기
처음엔 "응답 wrapper 를 사용합니다" 처럼 서술형으로 썼다. AI 는 이런 문장을 관례 소개 정도로 받아들이고 가끔 무시했다. "모든 응답은 아래 wrapper 를 반드시 사용. Controller 에서 직접 객체 반환 금지." 로 바꾸니 위반율이 눈에 띄게 줄었다. "해도 된다" vs "해야 한다" vs "하지 마라" — 톤이 AI 행동에 영향을 준다.

같은 요구(사용자 프로필 조회 API 추가)를 두 상태에서 비교

┌──────────────────────┬──────────────┬─────────────────┐

결과

02
Agent Layer · Subagent

spring-test-generator 서브에이전트

"반복 작업을 정형화해서 AI 에게 위임"
문제 원인

Spring 에서 테스트 코드 작성은 반복성이 높다. 서비스 클래스의 public 메서드 하나당 Happy / Edge / Exception / Branch 를 채우는 작업. 매번 수작업이면 시간은 많이 먹고 결과는 사람 컨디션에 따라 들쑥날쑥해진다. Claude 에게 "테스트 짜줘" 라고만 주면 매번 다른 스타일로 답이 왔다 — 어떤 날은 @SpringBootTest 로 풀 컨텍스트(느림), 어떤 날은 Mock 조차 없이 실제 객체(격리 안됨), given/when/then 주석을 붙였다 뺐다, 메서드 이름이 testCreateUser 와 createUser_whenEmailDuplicated_throwsException 을 왔다갔다. 일관성이 없으면 테스트 코드 자체가 유지보수 부담이 된다.

해결 과정

① 에이전트 정의 — 프런트매터로 호출 조건까지 명시

의사결정 근거 — 왜 description 에 <example> 까지 넣었나?
상위 에이전트(오케스트레이션 Claude)가 "언제 이 서브에이전트를 호출해야 하는가" 를 정확히 판단하려면 추상적 설명보다 구체적 사용 예시가 훨씬 효과적이었다. 4개 예시를 넣었더니 "테스트 관련 요청" 을 놓치지 않고 라우팅됐다. 프롬프트 엔지니어링에서 "예시" 가 "정의" 보다 강하다 는 원칙을 에이전트 설정에도 적용.

② 6단계 프로세스 — 흐름을 강제

Phase 1: Project Environment Analysis
  - build.gradle/pom.xml 에서 테스트 프레임워크 확인
  - 사용 가능한 라이브러리 카탈로그 (Mockito, AssertJ, Testcontainers)
  - application.yml 에서 테스트 프로파일 확인 (H2, Redis)
  - **기존 테스트 파일 패턴을 먼저 읽고 컨벤션 추출**

Phase 2: Source Code Analysis
  - @Service/@Repository/@Controller 판별 → 테스트 타입 결정
  - public 메서드 시그니처 나열, 의존성 → Mock 후보
  - 예외 처리, 트랜잭션 경계, 분기 분석

Phase 3: Test Type Selection and Generation
  - Unit:       @ExtendWith(MockitoExtension.class)
  - Controller: @WebMvcTest + MockMvc
  - Repository: @DataJpaTest + H2
  - 풀 컨텍스트: @SpringBootTest

Phase 4: Test Code Quality Standards
  - 이름 규칙: methodName_situation_expectedResult
    · createUser_whenEmailDuplicated_throwsException
  - given/when/then 구조 필수
  - AssertJ 우선, verify(mock, times(1)) 호출 검증

Phase 5: File Generation and Validation
  - 경로: src/test/java/{same package}
  - Pre-Save Checklist: 임포트, 메서드 존재, 이름 충돌, 패키지 선언

Phase 6: Summary Report
  - ✅ 생성 파일 경로 / 📋 클래스 수 / 📊 Happy/Edge/Exception/Branch 분포
  - ⚠️ 수동 검증 필요 항목
놓치기 쉬운 포인트 — "Phase 1 에서 기존 컨벤션을 먼저 읽어라"
이 순서가 없으면 에이전트가 "자기 생각대로 best practice" 를 적용해버려 기존 코드와 스타일이 어긋났다. ALWAYS read existing test files first if they exist 라는 문장을 Phase 1 에 못 박으니, 기존 프로젝트가 @Nested 를 쓰면 새 테스트도 @Nested 를, AssertJ 를 쓰면 AssertJ 를 자동으로 맞췄다. "AI 에게 '일관성 있게 해라' 라고 말하지 말고, '기존을 먼저 읽어라' 라고 말하라".

03
Skill Layer + Meta Evaluation

git-workflow 스킬 + A/B 자가 평가

"내가 만든 AI 도구를 내가 데이터로 검증한다"
문제 원인

서브에이전트 / 스킬을 여러 개 만들다보니 "이게 진짜 효과 있는 거 맞나?" 라는 본질적 의문이 생겼다. git-workflow 스킬은 Conventional Commits · 브랜치 네이밍 · Issue → Commit → PR 자동화를 담고 있는데, 스킬을 쓰는 것과 안 쓰는 것의 실제 차이가 감(感) 이 아니라 비교 가능한 형태로 보여야 했다. 그래야 (1) 쓸모없으면 빨리 버리고 (2) 쓸모있는 쪽을 더 정교화할 수 있다. "내가 만든 AI 도구를 내가 평가한다" — 개인 프로젝트에서도 필요한 검증 루프.

해결 과정

① 스킬 자체의 3단계 워크플로우

# .claude/skills/git-workflow/SKILL.md

## 1단계: 작업 계획 → Issue 생성 + 브랜치 제안
   - gh issue create 로 GitHub 에 직접 Issue 생성
   - type/stack/priority 라벨 자동 부여
   - Issue 번호 → 브랜치명 제안
     · feature/42-jwt-token-validation
     · bugfix/57-null-pointer-on-login
     · hotfix/103-critical-auth-bypass

## 2단계: 작업 중 → 커밋 메시지 생성
   - 파일 경로 → scope 자동 매핑 (14종 패턴)
     · *Controller.java   → controller
     · SecurityConfig / *Filter / *Provider → security
   - Conventional Commits 형식: type(scope): subject
   - subject 50자 이내 검증

## 3단계: 작업 완료 → PR 생성
   - 현재 브랜치 → base 자동 결정 (hotfix → main, 나머지 → dev)
   - gh pr create 로 체크리스트 포함 PR 본문 생성

## + references/conventions.md
   - type 9종, scope 20여종, Issue 라벨 4계층 전체 목록
   - 예제 커밋 메시지 6종
의사결정 근거 — 왜 conventions 를 별도 파일로 뺐나?
SKILL.md 상단(description, 1~3단계) 은 AI 가 매번 읽어야 하는 부분이고, conventions 는 참조용 레퍼런스다. 분리하면 (1) SKILL.md 는 간결 유지 (2) 레퍼런스가 필요한 순간에만 Claude 가 읽어서 컨텍스트 절약. 스킬 아키텍처도 "자주 쓰는 것 + 가끔 참조하는 것" 분리가 효과적이었다.

  • 스킬의 효과를 "체감" 이 아닌 "비교 가능한 아티팩트" 로 증명 — 8개 결과물이 저장소에 상시 보관
  • AI 도구 자체에 대한 개선 루프 세팅 — 다음 iteration 에서 뭘 고칠지 데이터로 드러남
  • Git 히스토리 Conventional Commits 100% 준수 — 자동 CHANGELOG · gh pr merge 자동화 연계 용이

Key Takeaways
01
명세가 AI 의 전부다 — CLAUDE.md 를 2단 계층으로 분리하고 "강한 명령형" 으로 작성하면, AI 의 첫 결과물이 "수정 3~4회 → 0~1회" 수준으로 개선된다. 프로젝트 관례는 머릿속에 두는 게 아니라 AI 가 읽을 수 있는 파일에 명문화해야 한다.
02
반복 작업은 서브에이전트로 위임한다 — 테스트 생성처럼 구조가 뚜렷한 작업은 6-phase 프로세스로 정형화한 spring-test-generator 에게 맡기고, 사람은 설계 · 리뷰에 집중한다. <example> 을 description 에 넣어 호출 정확도를 프롬프팅 원리로 끌어올렸다.
03
내가 만든 AI 도구도 A/B 로 검증한다 — with_skill vs without_skill 4개 시나리오를 iteration-1/ 디렉토리에 남겼다. type 정확도 · 중복 표기 · body 품질 등 구체적 지표로 스킬 효과를 증명 — 감이 아니라 데이터로 반복 개선이 가능해진 게 진짜 차별점이다.
