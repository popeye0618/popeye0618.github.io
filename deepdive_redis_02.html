실시간 게임
랭킹 시스템

100만 플레이어의 점수를 실시간으로 집계하고 순위를 조회합니다. DB의 ORDER BY 한계를 직접 확인하고, Sorted Set의 O(log N) 특성으로 대용량 트래픽에서도 일정한 응답시간을 보장합니다.

Java 21 Spring Boot 4.x Redis 8.0 Redisson 4.0.0 MySQL 8.0 JMeter 5.6.3 Docker Compose
Architecture

전체 시스템 구조

점수 등록은 Redis Sorted Set에 즉시 반영되고 DB에 비동기로 저장됩니다. 시즌 종료는 TTL 만료 이벤트로 자동 처리됩니다.

Core Design

핵심 설계 — 왜 DB가 아닌 Redis인가

실시간 랭킹을 DB로 구현하면 요청마다 대용량 정렬 연산이 발생합니다. Redis Sorted Set은 삽입 시점에 이미 정렬이 완료됩니다.

DB 기반

MySQL ORDER BY

-- 랭킹 조회마다 실행 SELECT user_id, score FROM game_scores WHERE season_id = 1 ORDER BY score DESC LIMIT 100; -- 플레이어 100만 명 → 매 요청마다 정렬 -- 초당 수천 번 호출 → DB 병목
Redis 기반

Sorted Set ZREVRANGE

-- 삽입 시 자동 정렬 완료 ZADD leaderboard:season:1 5000 "userId:10" -- 조회는 이미 정렬된 결과에서 ZREVRANGE leaderboard:season:1 0 99 WITHSCORES -- 항상 O(log N + M) -- 플레이어 수 무관하게 일정
Thought Process

사고 흐름 — 기능 구현부터 안정성까지

단순 랭킹 조회에서 시작해 Rate Limiting, 범위 쿼리, 시즌 종료까지 각 단계에서 "다음 문제는 무엇인가"를 먼저 정의했습니다.

Version 1 — 기반 구현
실시간 점수 등록 · 랭킹 조회
"ZADD로 점수를 넣으면 Redis가 자동으로 정렬해줍니다. 이 단순함이 핵심입니다. 조회는 이미 정렬된 결과에서 꺼내기만 하면 됩니다."

Version 2 — 어뷰징 방지
Sliding Window Rate Limiting — 비원자적 구현의 함정
"같은 유저가 1초에 수백 번 점수를 올리면 어떻게 될까요. Fixed Window는 경계 지점에서 뚫립니다. ZREMRANGEBYSCORE → ZCARD → ZADD를 분리 실행하면 그 사이에 다른 요청이 끼어듭니다."
V1 → V2 의사결정

비원자적 구현(V1)에서 동시 20명 요청 시 한도 10개인데 18개가 통과되는 Race Condition을 JMeter로 재현했습니다. Lua Script로 세 명령어를 원자적으로 묶어(V2) 정확히 10개만 통과하도록 해결했습니다. Project 1의 Lua Script 패턴을 Rate Limiting 도메인에 그대로 적용한 케이스입니다.

Version 3 — 범위 쿼리
ZRANGEBYSCORE · 퍼센타일 계산
"'상위 30% 커트라인 점수가 얼마인가'를 DB로 구현하면 서브쿼리가 필요합니다. Redis는 ZREVRANK + ZCARD 두 번으로 해결됩니다."

ZRANGEBYSCORE로 특정 점수 구간 플레이어를 조회하고, ZREVRANK ÷ ZCARD로 상위 퍼센타일을 계산합니다. 모든 연산이 O(log N) 이하로 처리됩니다.

V4
Version 4 — 라이프사이클
TTL + Keyspace Notification — 시즌 종료 자동화
"@Scheduled로 매 분마다 시즌 종료 여부를 체크하면 불필요한 DB 조회가 발생합니다. Redis가 TTL 만료를 직접 알려준다면 정확한 시각에 이벤트 기반으로 처리할 수 있습니다."

시즌 생성 시 season:expire:{id} 키에 TTL을 설정합니다. 만료 순간 Redis가 이벤트를 발행하고 리스너가 최종 랭킹 스냅샷 저장 → Redis 키 정리 → 시즌 상태 변경을 순서대로 처리합니다.

Key Takeaways

핵심 인사이트

01

자료구조가 곧 설계다

Sorted Set을 선택한 순간 랭킹 조회, 범위 쿼리, 퍼센타일 계산이 모두 해결됩니다. 올바른 자료구조 선택이 코드 복잡도를 결정합니다.

02

같은 문제, 다른 도메인

Rate Limiting의 Race Condition은 Project 1의 재고 Race Condition과 동일한 구조입니다. Lua Script 원자적 처리 패턴을 다른 도메인에 적용했습니다.

03

이벤트 기반 vs 폴링

시즌 종료를 스케줄러로 처리하면 매 분마다 불필요한 DB 조회가 발생합니다. TTL 만료 이벤트로 처리하면 정확한 시각에 한 번만 실행됩니다.
