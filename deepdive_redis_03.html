V1
Version 1 — 기반 구현
Redis Stream 기본 발행/소비
"XADD로 쌓고 XREAD로 꺼내는 구조는 단순합니다. 그런데 두 소비자가 동시에 XREAD를 하면 같은 메시지를 둘 다 받습니다. 그리고 처리 중 서버가 죽으면 그 메시지는 누가 처리하나요?"

append-only 구조로 메시지를 적재하고 @Scheduled 폴링으로 소비하는 기본 흐름을 구현했습니다. 이 단계의 목적은 흐름을 이해하고 한계를 직접 확인하는 것이었습니다.

V3
Version 3 — 실시간성 확보
Pub/Sub → WebSocket — 브라우저까지 즉시 전달
"Stream은 내구성은 있지만 폴링 기반이라 실시간이 아닙니다. Consumer가 DB에 저장하는 시점에 Pub/Sub 채널에도 발행하면, 연결된 WebSocket으로 즉시 전달할 수 있습니다."

Consumer 처리 완료 시점에 PUBLISH notification:pubsub:{userId}를 호출합니다. PatternTopic("notification:pubsub:*")로 전체 유저 채널을 한 번에 구독하고, userId를 채널 이름에서 파싱해 해당 WebSocket으로 라우팅합니다.

Key Takeaways

핵심 인사이트

01

두 기술의 역할을 분리한다

Stream은 내구성, Pub/Sub는 실시간성. 각자 잘하는 것을 맡기면 복잡한 요구사항도 단순하게 해결됩니다.

02

XACK가 없으면 처리된 게 아니다

메시지를 읽은 것과 처리한 것은 다릅니다. XACK를 보내기 전까지 Redis는 그 메시지를 미처리로 간주합니다.

03

PEL이 신뢰성의 기반이다

PEL 없이는 장애 복구가 불가능합니다. Kafka의 Offset 커밋과 동일한 역할을 Redis Stream의 PEL + XACK가 담당합니다.
